## Мета роботи
- Вивчення принципів побудови REST API.
- Набуття практичних навичок розробки серверного застосунку з використанням платформи Node.js і фреймворку Express.
- Реалізація механізмів реєстрації та авторизації користувачів.
- Забезпечення валідації вхідних даних та обробки помилок.
- Організація захищеного доступу до ресурсів із використанням JWT-токенів і системи ролей користувачів.

### Завдання 8, 11. Зберігати користувачів у базі та додати роль користувача (admin/user)
За допомогою ORM Sequelize було створено нову модель `User` для зберігання даних користувачів у базі MySQL. В моделі передбачено поле `role`, яке за замовчуванням отримує значення `user`.

**Програмний код (`models/User.js`):**
```javascript
const { DataTypes } = require("sequelize");
const sequelize = require("../config/database");

const User = sequelize.define("User", {
    email: {
        type: DataTypes.STRING,
        allowNull: false,
        unique: true,
        validate: { isEmail: true }
    },
    password: { type: DataTypes.STRING, allowNull: true },
    role: { type: DataTypes.STRING, defaultValue: "user" },
    refreshToken: { type: DataTypes.STRING, allowNull: true },
    isVerified: { type: DataTypes.BOOLEAN, defaultValue: false }
}, {
    timestamps: true,
});

module.exports = User;
```

### Завдання 13, 14. Додати логування помилок та обмежити кількість спроб входу
Для відстеження помилок та моніторингу HTTP-запитів було підключено логер `morgan`. Для захисту API від атак типу Brute-force встановлено обмеження (`express-rate-limit`): максимум 3 спроби входу на 1 хвилину.

**Програмний код (`server.js`):**
```javascript
// Завдання 13: Логування запитів
app.use(morgan('dev'));

// Завдання 14: Обмеження кількості спроб входу
const loginLimiter = rateLimit({
    windowMs: 1 * 60 * 1000, // 1 хвилина
    max: 3, // 3 спроби
    message: { error: "Забагато спроб входу. Зачекайте 1 хвилину." }
});
```

### Завдання 15. Додати middleware для перевірки токена
Створено функцію `authenticateToken`, яка перевіряє наявність та валідність JWT-токена в заголовку `Authorization`.

**Програмний код (`server.js`):**
```javascript
const authenticateToken = (req, res, next) => {
    const authHeader = req.headers['authorization'];
    const token = authHeader && authHeader.split(' ')[1];
    
    if (!token) return res.status(401).json({ error: "Токен відсутній." });

    jwt.verify(token, SECRET_KEY, (err, user) => {
        if (err) return res.status(403).json({ error: "Сесія недійсна. Увійдіть знову." });
        req.user = user;
        next();
    });
};
```

### Завдання 2, 3, 7. Реалізувати реєстрацію з валідацією та підтвердженням пароля
Реалізовано маршрут `/register`. Здійснюється сувора перевірка: наявність усіх полів, збіг паролів (завдання 7) та перевірка надійності пароля регулярним виразом (завдання 3). Пароль хешується перед збереженням.

**Програмний код (`server.js`):**
```javascript
app.post("/api/auth/register", async (req, res) => {
    try {
        const { email, password, confirmPassword } = req.body;
        
        // Завдання 3: Валідація та обробка помилок
        if (!email || !password || !confirmPassword) return res.status(400).json({ error: "Всі поля обов'язкові" });
        
        const passwordRegex = /^(?=.*[A-Z])(?=.*\d)[a-zA-Z\d]{6,}$/;
        if (!passwordRegex.test(password)) {
            return res.status(400).json({ error: "Пароль має містити мін. 6 символів, 1 велику літеру та 1 цифру." });
        }
        
        // Завдання 7: Підтвердження пароля
        if (password !== confirmPassword) return res.status(400).json({ error: "Паролі не співпадають" });

        const existingUser = await User.findOne({ where: { email } });
        if (existingUser) return res.status(400).json({ error: "Користувач вже існує" });

        const hashedPassword = await bcrypt.hash(password, 10);
        const newUser = await User.create({ email, password: hashedPassword });
        
        res.status(201).json({ message: "Реєстрація успішна!", userId: newUser.id });
    } catch (error) {
        res.status(500).json({ error: "Помилка сервера при реєстрації" });
    }
});
```

### Завдання 2, 12. Авторизація та реалізація Refresh Token
При логіні перевіряється захешований пароль. Присвоюються два токени: `accessToken` (на 15 хв) та `refreshToken` (зберігається в базі, діє 7 днів). 

**Програмний код (`server.js`):**
```javascript
app.post("/api/auth/login", loginLimiter, async (req, res) => {
    const { email, password } = req.body;
    const user = await User.findOne({ where: { email } });
    
    if (!user || !user.password) return res.status(400).json({ error: "Користувача не знайдено." });

    const isMatch = await bcrypt.compare(password, user.password);
    if (!isMatch) return res.status(400).json({ error: "Невірний пароль" });

    const accessToken = jwt.sign({ id: user.id, email: user.email, role: user.role }, SECRET_KEY, { expiresIn: '15m' });
    
    // Завдання 12: Refresh token
    const refreshToken = jwt.sign({ id: user.id }, REFRESH_SECRET_KEY, { expiresIn: '7d' });
    user.refreshToken = refreshToken;
    await user.save();

    res.json({ message: "Вхід успішний", accessToken });
});
```

### Завдання 4, 10. Захищений маршрут та оновлення профілю
Створено маршрут для отримання даних поточного користувача. Реалізовано функцію оновлення електронної пошти (із попередньою перевіркою токена).

    const user = await User.findByPk(req.user.id, { attributes: ['email', 'role', 'isVerified'] });

### Завдання 16, 17. Зміна пароля та видалення користувача
Реалізовано метод для зміни пароля, який вимагає підтвердження старого пароля (через `bcrypt.compare`). Додано метод `DELETE`, що назавжди видаляє запис користувача з бази даних.

**Програмний код (`server.js`):**
```javascript
// Завдання 16: Зміна пароля
app.put("/api/auth/change-password", authenticateToken, async (req, res) => {
    const { oldPassword, newPassword } = req.body;
    const user = await User.findByPk(req.user.id);

    const isMatch = await bcrypt.compare(oldPassword, user.password);
    if (!isMatch) return res.status(400).json({ error: "Невірний поточний пароль" });

    user.password = await bcrypt.hash(newPassword, 10);
    await user.save();
    res.json({ message: "Пароль змінено!" });
});

// Завдання 17: Видалення користувача
app.delete("/api/auth/profile", authenticateToken, async (req, res) => {
    await User.destroy({ where: { id: req.user.id } });
    res.json({ message: "Акаунт успішно видалено." });
});
```

### Завдання 9. Реалізувати logout
Маршрут для виходу з системи: сервер очищує `refreshToken` в базі даних, а клієнтська частина видаляє `accessToken` із `localStorage`.

**Програмний код (`server.js`):**
```javascript
app.post("/api/auth/logout", authenticateToken, async (req, res) => {
    const user = await User.findByPk(req.user.id);
    if(user) {
        user.refreshToken = null; 
        await user.save();
    }
    res.json({ message: "Ви вийшли з системи" });
});
```

### Завдання 6. Проаналізувати отримані результати
Під час тестування розробленого REST API через браузер та Postman було успішно перевірено повний життєвий цикл користувача в системі:
1. Метод `POST /api/auth/register` коректно обробляє вхідні дані, здійснює перевірку надійності пароля (мінімум 6 символів, цифри та великі літери) та успішно хешує його в базі даних.
2. Метод `POST /api/auth/login` успішно автентифікує користувача, генерує безпечний `JWT-токен` та надійно обробляється лімітером (при перевищенні спроб видає статус HTTP 429 Too Many Requests).
3. Передача токена в заголовку `Authorization: Bearer <token>` дозволила протестувати доступ до захищених маршрутів (наприклад, `GET /api/auth/profile`). Middleware `authenticateToken` ефективно відхиляє запити без токена (HTTP 401) або з недійсним токеном (HTTP 403).
4. Методи `PUT` та `DELETE` для оновлення профілю і видалення акаунту коректно ідентифікують користувача (ID розшифровується з токена) і виконують дії в базі MySQL. Механізм імітації підтвердження email довів гнучкість системи до додавання нових статусів користувача.

## Висновки
Під час виконання лабораторної роботи я розробила повноцінний та безпечний REST API для веб-застосунку на платформі Node.js із використанням фреймворку Express. Я засвоїла принципи stateless-архітектури, налаштувала маршрутизацію та обробку HTTP-запитів різних типів (GET, POST, PUT, DELETE). 
Важливим етапом стало впровадження сучасних механізмів безпеки: хешування паролів за допомогою `bcryptjs`, реалізація надійної системи авторизації на основі JWT-токенів (з використанням стратегії Access/Refresh токенів), а також захист маршрутів за допомогою кастомних middleware. Окрім цього, я реалізувала сувору валідацію вхідних даних та коректну обробку помилок із відповідними HTTP-статусами, що робить створений API стійким до некоректних або зловмисних дій користувачів. Інтеграція розробленого бекенду з клієнтським інтерфейсом підтвердила правильність усіх застосованих архітектурних рішень.

## Контрольні питання
1. **Що таке REST API та які його основні принципи?**
   *REST API (Representational State Transfer) - це архітектурний стиль взаємодії клієнта і сервера через протокол HTTP. Його основними принципами є чіткий поділ на клієнта та сервер, відсутність збереження стану між запитами (Stateless), можливість кешування відповідей для підвищення швидкодії, а також використання єдиного уніфікованого інтерфейсу для доступу до ресурсів через унікальні URL.*
2. **Які HTTP-методи використовуються в REST API і для чого вони призначені?**
   *У REST API використовуються стандартні HTTP-методи, кожен з яких виконує свою функцію: GET призначений для отримання даних про ресурс, POST - для створення нового об’єкта, PUT використовується для повного оновлення існуючого ресурсу, PATCH дозволяє внести часткові зміни (наприклад, змінити лише email), а DELETE — видалити ресурс із системи*
3. **Що таке клієнт-серверна архітектура?**
   *Клієнт-серверна архітектура передбачає розподіл обов’язків, де клієнт виступає в ролі інтерфейсу користувача, а сервер відповідає за бізнес-логіку та обробку даних. Це дозволяє розробляти та масштабувати обидві частини незалежно одна від одної, що спрощує підтримку складних систем.*
4. **Що означає принцип Stateless у REST?**
   *Принцип Stateless означає, що сервер не пам’ятає попередніх запитів клієнта і не зберігає стан його сесії. Кожен окремий HTTP-запит повинен містити в собі абсолютно всю інформацію, необхідну серверу для ідентифікації користувача та виконання завдання.*
5. **Для чого використовується фреймворк Express у Node.js?**
   *Фреймворк Express є легким інструментом для Node.js, який значно полегшує розробку серверної частини веб-застосунків. Він надає готові засоби для налаштування маршрутів (routing), обробки HTTP-запитів, роботи з JSON-даними та підключення проміжних шарів (middleware) для реалізації логіки програми*
6. **Що таке реєстрація користувача в інформаційній системі?**
   *Реєстрація користувача - це процес внесення даних нової особи до бази даних інформаційної системи. Вона включає введення email та пароля, обов'язкову перевірку коректності цих даних (валідацію), хешування пароля для безпеки та створення нового запису в таблиці користувачів.*
7. **Що таке авторизація та чим вона відрізняється від автентифікації?**
   *Автентифікація - це процедура перевірки особистості користувача (наприклад, за допомогою звірки введеного пароля із захешованим значенням у базі). Авторизація ж визначає права доступу користувача до певних розділів системи або дій, наприклад, чи має він право видаляти товари або переглядати адміністративну панель.*
8. **Для чого використовується JWT (JSON Web Token)?**
   *JWT (JSON Web Token) використовується для безпечної передачі інформації між клієнтом і сервером у форматі JSON-об'єкта. Його головна мета - автентифікація: після успішного входу сервер видає токен, який клієнт зберігає та додає до кожного наступного запиту, підтверджуючи своє право на доступ до захищених ресурсів.*
9. **З яких частин складається JWT-токен?**
   *JWT-токен складається з трьох основних частин, розділених крапками: заголовка (Header), Payload (корисного навантаження з даними про користувача, такими як id або роль) та криптографічного підпису (Signature), який гарантує цілісність даних і захищає їх від підробки.*
10. **Для чого потрібно хешування паролів і яка бібліотека використовується в роботі?**
    *Хешування паролів необхідне для захисту персональних даних: у базі зберігається не сам пароль, а його незворотний математичний відбиток, що робить крадіжку реальних паролів неможливою навіть у разі зламу БД. Для виконання цього завдання у роботі була використана бібліотека bcryptjs.*
11. **Що таке валідація даних і навіщо вона потрібна?**
    *Валідація даних - це перевірка вхідної інформації (email, довжина пароля) на відповідність встановленим вимогам перед її обробкою сервером. Вона потрібна для запобігання помилкам, захисту бази даних від некоректних записів та підвищення загальної безпеки системи.*
12. **Які основні HTTP-коди стану використовуються для обробки помилок?**
    *Для обробки помилок у REST API використовуються HTTP-коди стану: 400 Bad Request (неправильні вхідні дані), 401 Unauthorized (користувач не авторизований), 403 Forbidden (доступ заборонено через недостатні права), 404 Not Found (ресурс не знайдено) та 500 Server Error (внутрішня помилка сервера).*
13. **Що таке middleware та яку роль він виконує в Express?**
    *Middleware - це спеціальні функції-посередники в Express, які виконуються під час обробки запиту. Вони відіграють роль фільтрів: можуть логувати дії користувачів, перевіряти JWT-токени на валідність або обробляти помилки, передаючи керування далі по ланцюжку за допомогою функції next().*
14. **Для чого потрібен захищений маршрут у REST API?**
    *Захищений маршрут потрібен для того, щоб обмежити доступ до певних частин API тільки для авторизованих користувачів. Наприклад, переглянути особистий кабінет або змінити пароль може тільки той користувач, який надав серверу правильний токен доступу.*
15. **Що таке роль користувача (user/admin) і як вона впливає на доступ до ресурсів?**
    *Роль користувача (user/admin) визначає рівень його привілеїв у системі. Вона безпосередньо впливає на доступ: звичайний користувач може керувати лише власним профілем, тоді як адміністратор має повні права на редагування каталогу товарів, категорій та управління обліковими записами інших користувачів.*
