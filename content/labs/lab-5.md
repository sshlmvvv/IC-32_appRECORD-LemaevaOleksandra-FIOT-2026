# Лабораторна робота № 5: «Безпека та продуктивність серверних додатків. Безпека Node.js-додатків. Оптимізація запитів і кешування. Тестування API»

## Мета роботи
- Забезпечити базову безпеку Node.js-додатків від поширених веб-вразливостей
- Оптимізувати продуктивність REST API за допомогою сучасних архітектурних рішень
- Використати in-memory кешування для зменшення навантаження на базу даних та сервер
- Навчитися тестувати API за допомогою сучасних інструментів навантажувального тестування
- Здійснити глибокий аналіз продуктивності backend-застосунків до та після оптимізації

## Хід роботи та виконання завдань

### Завдання 1. Забезпечення базової безпеки API та валідація
Для захисту серверного застосунку від поширених вразливостей (таких як XSS, clickjacking, DDoS-атаки та брутфорс) було встановлено та налаштовано спеціальні пакети: `helmet`, `express-rate-limit` та `express-validator`

~~~text
npm install helmet express-rate-limit express-validator
~~~
![Скріншот: Встановлення пакетів безпеки](/assets/labs/lab-5/npm_install_security.png)

**1. Захист HTTP-заголовків (Helmet):**
Бібліотека `helmet` автоматично приховує технологічний стек сервера (заголовок `X-Powered-By`) та додає строгі політики безпеки. У коді це підключається як глобальне middleware.
~~~javascript
const helmet = require("helmet");
app.use(helmet({ contentSecurityPolicy: false }));
~~~
![Скріншот: Заголовки безпеки Helmet у Postman](/assets/labs/lab-5/helmet_security_headers.png)

**2. Захист від перевантажень (Rate Limiting):**
Щоб запобігти зловживанням API (наприклад, DDoS-атакам або підбору паролів), було створено лімітер. Для маршрутів авторизації встановлено жорсткий ліміт — не більше 10 запитів на 1 хвилину з однієї IP-адреси. При перевищенні ліміту сервер блокує клієнта і повертає помилку 429.
~~~javascript
const loginLimiter = rateLimit({
    windowMs: 1 * 60 * 1000, // 1 хвилина
    max: 10,                 // Максимум 10 спроб
    message: { error: "Забагато спроб входу. Зачекайте 1 хвилину." },
});
app.use("/api/auth/login", loginLimiter, /* ... */);
~~~
![Скріншот: Спрацювання захисту Rate Limit](/assets/labs/lab-5/rate_limit_triggered.png)

**3. Валідація вхідних даних (Express-Validator):**
Щоб у базу даних не потрапляло некоректне "сміття" та для захисту від ін'єкцій, реалізовано сувору перевірку даних. Під час реєстрації сервер перевіряє формат email, а також складність пароля (мінімум 6 символів, обов'язкова наявність великої літери та цифри).
~~~javascript
app.post("/api/auth/register", [
    body("email").isEmail().withMessage("Некоректний формат email"),
    body("password").isLength({ min: 6 })
      .matches(/^(?=.*[A-Z])(?=.*\d)[a-zA-Z\d]{6,}$/)
      .withMessage("Пароль має містити 1 велику літеру(латиниця) та 1 цифру.")
], async (req, res, next) => {
    const errors = validationResult(req);
    if (!errors.isEmpty()) {
        return res.status(400).json({ errors: errors.array() });
    }
    // ... логіка реєстрації
});
~~~
![Скріншот: Помилка валідації пароля та email](/assets/labs/lab-5/express_validator_error.png)

### Завдання 2. Реалізація JWT-автентифікації
Традиційні сесії перевантажують пам'ять сервера, тому застосунок було переведено на архітектуру без збереження стану (stateless) за допомогою JSON Web Tokens (JWT). Після успішної перевірки credentials (email та хешованого пароля), сервер генерує `accessToken`.

Клієнт повинен передавати цей токен у заголовку `Authorization: Bearer <token>` для доступу до захищених маршрутів (наприклад, перегляду профілю).
~~~javascript
// Генерація токена при успішному логіні
const accessToken = jwt.sign(
    { id: user.id, email: user.email, role: user.role },
    SECRET_KEY,
    { expiresIn: "15m" }
);
res.json({ message: "Вхід успішний", accessToken });
~~~

### Завдання 3. Документація API (Swagger)
Для зручного тестування API та полегшення інтеграції з Frontend-клієнтами було розроблено інтерактивну документацію за стандартом OpenAPI 3.0. Для цього підключено пакети `swagger-ui-express` та `swagger-jsdoc`.

Документація доступна за маршрутом `/api-docs`. У ній детально описані всі доступні endpoint-и, очікувані формати тіла запиту (Request Body) та можливі статуси відповідей.
~~~javascript
const swaggerOptions = {
  definition: {
    openapi: "3.0.0",
    info: { title: "Flower Shop API", version: "1.0.0" },
    components: {
      securitySchemes: {
        bearerAuth: { type: "http", scheme: "bearer", bearerFormat: "JWT" },
      },
    },
    // ... опис шляхів (paths)
  },
  apis: [],
};
const swaggerDocs = swaggerJsdoc(swaggerOptions);
app.use("/api-docs", swaggerUi.serve, swaggerUi.setup(swaggerDocs));
~~~

### Завдання 5. Оптимізація запитів та Redis-кешування
Читання даних із жорсткого диска (бази даних MySQL) є "вузьким місцем" у продуктивності будь-якого API. Для вирішення цієї проблеми було впроваджено in-memory кешування за допомогою **Redis**. Оптимізовано маршрут отримання списку букетів (`GET /api/bouquets`).

**Алгоритм роботи оптимізованого маршруту:**
1. При надходженні запиту сервер звертається до оперативної пам'яті Redis.
2. Якщо дані є (Кеш-хіт) — вони миттєво віддаються клієнту.
3. Якщо даних немає (Кеш-місс) — сервер виконує повільний SQL-запит до MySQL, після чого зберігає результат у Redis на 60 секунд (`EX: 60`), щоб наступні запити оброблялися швидко.
~~~javascript
app.get("/api/bouquets", async (req, res, next) => {
  const CACHE_KEY = "bouquets_list";
  try {
    const cachedData = await redisClient.get(CACHE_KEY);
    if (cachedData) {
      console.log('--- [REDIS] ДАНІ ВЗЯТО З КЕШУ ---');
      return res.json(JSON.parse(cachedData));
    }

    console.log('--- [MySQL] КЕШ ПОРОЖНІЙ, ЗАПИТ ДО БД ---');
    const bouquets = await Bouquet.findAll({ include: Category });
    await redisClient.set(CACHE_KEY, JSON.stringify(bouquets), { EX: 60 });
    res.json(bouquets);
  } catch (error) { next(error); }
});
~~~

#### Аналіз та обґрунтування результатів тестування
Нижче наведено фінальний звіт Artillery (`Summary report`). Перед тим як переглянути його, варто відзначити ключові показники, які підтверджують успішність виконаної роботи:

1. **Миттєва швидкість (Median: 10.9 ms):** Медіанний час обробки запиту становить майже 11 мілісекунд, а найшвидший (`min`) — всього 3 мс. Це феноменально швидкий показник для Node.js сервера під навантаженням. Такий результат був би фізично неможливим при постійному зверненні до диска бази даних. Це прямий доказ того, що **Redis-кеш працює** і миттєво віддає масив даних безпосередньо з оперативної пам'яті.
2. **Ефективна робота системи безпеки (http.codes.429: 222):** У звіті зафіксовано 222 відповіді з кодом помилки `429 Too Many Requests`. Це не свідчить про збій сервера, а навпаки — є доказом того, що підключений у Завданні 1 `express-rate-limit` відпрацював ідеально. Оскільки Artillery "бомбардував" сервер запитами, захист виявив аномалію (перевищення ліміту у 100 запитів) і почав блокувати зайві звернення. Це захистило сервер від падіння (відмови в обслуговуванні).
3. **Стабільність (vusers.failed: 0):** Незважаючи на агресивне навантаження, кількість впавших з'єднань або критичних помилок сервера (`500 Internal Server Error`) дорівнює нулю.

## Висновки
У ході виконання лабораторної роботи №5 я глибоко вивчила питання безпеки, оптимізації та масштабування backend-застосунків. 

Було реалізовано багаторівневий захист: вхідні дані проходять жорстку перевірку (express-validator), HTTP-заголовки захищені від ін'єкцій та сніффінгу (Helmet), а сервер надійно прикритий від спаму та DDoS-атак лімітером запитів. Перехід з традиційних сесій на JWT-автентифікацію дозволив зробити API stateless, що є сучасним стандартом для масштабованих систем. Також було згенеровано зручну OpenAPI-документацію через Swagger.

Ключовим етапом стала контейнеризація всього стеку технологій (Node.js, MySQL, Redis) за допомогою Docker. Це дозволило безболісно розгорнути in-memory сховище Redis та інтегрувати його у маршрутизатор API. Фінальне навантажувальне тестування (Artillery) беззаперечно підтвердило правильність обраних рішень: завдяки кешуванню швидкість відповіді сервера зменшилася до ~11 мс, а впроваджені модулі безпеки ефективно відбили спробу імітованої DoS-атаки, не допустивши падіння бази даних.

## Контрольні запитання

**1. Що таке Helmet у Node.js?**
Helmet — це набір middleware функцій для Express-додатків, які автоматично встановлюють безпечні HTTP-заголовки. Це допомагає захистити сервер від поширених веб-вразливостей, таких як XSS (міжсайтовий скриптинг), Clickjacking, атаки через MIME-сніффінг, а також приховує заголовок `X-Powered-By`, ускладнюючи зловмисникам ідентифікацію технологічного стеку.

**2. Для чого використовується rate limiting?**
Rate limiting використовується для обмеження кількості запитів до API від одного клієнта (IP-адреси) за певний проміжок часу. Це критично важливо для захисту сервера від цілеспрямованих DDoS-атак, брутфорсу (підбору паролів), а також для справедливого розподілу ресурсів сервера між усіма користувачами.

**3. Які переваги кешування?**
Кешування забезпечує значне зменшення навантаження на базу даних та процесор сервера. Воно суттєво пришвидшує час відповіді клієнту (зменшує latency), оскільки дані дістаються з надшвидкої оперативної пам'яті, а не з диска. Це також значно покращує масштабованість системи при високих навантаженнях.

**4. Чим відрізняється unit testing від integration testing?**
Unit testing (модульне тестування) перевіряє працездатність окремих, ізольованих одиниць коду (наприклад, однієї функції валідації) без підключення до баз даних або мережі. Integration testing (інтеграційне тестування) перевіряє, як кілька різних модулів (наприклад, Express-маршрут, база даних та Redis) взаємодіють між собою як єдина система.

**5. Для чого потрібна валідація даних?**
Валідація необхідна для перевірки коректності, формату та безпеки вхідних даних. Це перший рубіж захисту, який не дозволяє шкідливому коду або ін'єкціям (SQL/NoSQL) потрапити до бази даних, гарантуючи цілісність бізнес-логіки та захищаючи систему від непередбаченої поведінки.

**6. Що таке JWT?**
JWT (JSON Web Token) — це відкритий стандарт безпечної передачі інформації у вигляді зашифрованого JSON-об'єкта. У веб-додатках він використовується для stateless-автентифікації. JWT зберігається на стороні клієнта, не перевантажуючи пам'ять сервера, а його цілісність гарантується криптографічним підписом, який неможливо підробити без "секретного ключа" сервера.

**7. Які типи кешування існують?**
Основні типи кешування: In-memory cache (кеш усередині процесу Node.js), розподілені кеш-сховища (наприклад, Redis або Memcached), CDN-кешування (збереження статики на проміжних серверах ближче до користувача) та Browser cache (збереження даних безпосередньо у веб-браузері клієнта).

**8. Які існують способи оптимізації API?**
Способи оптимізації включають: індексацію бази даних (для швидкого пошуку), впровадження кешування (щоб уникнути зайвих запитів до БД), пагінацію (розбиття великих списків на малі сторінки), асинхронну обробку важких задач у фоні, стиснення відповідей (наприклад, Gzip/Brotli для зменшення розміру payload) та оптимізацію складних SQL/NoSQL-запитів.
